# 仮想通貨自動売買システム「一攫千金くん」詳細設計書 v0.1

## 1. 本書の目的

本書は、基本設計書をもとに、実装に入れる粒度までシステム内部の構成、処理内容、データベース構造、売買ロジック、通知仕様、エラー処理を具体化するものである。

本書は初期開発版のたたき台であり、実装しながら調整する前提とする。

---

## 2. システム概要

### 2.1 システム名

仮想通貨自動売買システム  
**一攫千金くん**

### 2.2 目的

仮想通貨の価格情報および複数時間足のシグナルをもとに、自動で売買判断を行う。

売買発生時にはLINE通知を行い、売却時には損益、総資産、取引結果を通知する。

将来的には、GUI、シグナル評価、自動パラメータ調整、対象通貨拡張を行い、利益最大化を目指す。

## 7. 売買ロジック詳細

### 7.1 初期方針

初期版では、複雑なAI判断は行わず、ルールベースで売買する。

複数時間足の方向性が一致した場合にエントリー候補とする。

### 7.2 使用時間足

| 時間足 | 用途 |
|---|---|
| 1分足 | 短期エントリー確認 |
| 5分足 | メイン判断 |
| 15分足 | 上位足方向確認 |
| 1時間足 | 大きなトレンド確認 |

初期実装では、5分足と15分足を主軸にする。

### 7.3 使用指標

| 指標 | 用途 |
|---|---|
| 移動平均線 | トレンド方向確認 |
| RSI | 買われすぎ・売られすぎ確認 |
| MACD | トレンド転換確認 |
| 出来高 | 急変動・勢い確認 |
| 価格変化率 | 短期急騰・急落検知 |

### 7.4 買いエントリー条件

以下をすべて満たす場合、買いエントリー候補とする。

```text
1. 15分足の短期移動平均線が長期移動平均線を上回っている
2. 5分足の短期移動平均線が長期移動平均線を上回っている
3. RSIが買われすぎ水準ではない
4. MACDが上向き
5. 直近出来高が一定以上
6. 現在ポジションを保有していない
7. システム停止条件に該当していない
```

### 7.5 買い注文

| 項目 | 内容 |
|---|---|
| 注文種別 | 成行買い |
| 注文金額 | 初期設定値に従う |
| 対象通貨 | 初期はBTC |
| 実行後処理 | 約定結果取得、DB保存、LINE通知 |

要確認：1回あたりの注文金額。  
仮置き：5,000円。

### 7.6 利確条件

買いポジション保有中、現在価格が平均取得価格から一定割合以上上昇した場合、利確指値注文を出す。

```text
利確価格 = 平均取得価格 × (1 + 利確率)
```

| 項目 | 仮置き値 |
|---|---|
| 利確率 | 0.5% |
| 注文種別 | 指値売り |

### 7.7 損切り条件

買いポジション保有中、現在価格が平均取得価格から一定割合以上下落した場合、成行売りで損切りする。

```text
損切り価格 = 平均取得価格 × (1 - 損切り率)
```

| 項目 | 仮置き値 |
|---|---|
| 損切り率 | 0.7% |
| 注文種別 | 成行売り |

### 7.8 売却後処理

```text
1. 約定価格取得
2. 取引手数料取得
3. 損益計算
4. 総資産取得
5. trade履歴登録
6. positionをクローズ
7. LINE通知送信
```

## 8. リスク管理設計

### 8.1 停止条件

| 条件 | 内容 |
|---|---|
| API接続エラー連続発生 | 一定回数以上で停止 |
| 注文失敗連続発生 | 一定回数以上で停止 |
| 1日の損失上限到達 | 当日取引停止 |
| 連続損切り回数超過 | 当日取引停止 |
| 残高不足 | 取引停止 |
| 想定外ポジション検知 | 取引停止 |
| 手動停止フラグON | 即時停止 |
| 急変動検知 | 一時停止 |

### 8.2 仮置きリスク設定

| 項目 | 仮置き値 |
|---|---|
| 1日最大損失率 | 総資産の3% |
| 連続損切り停止回数 | 3回 |
| API連続失敗停止回数 | 5回 |
| 注文連続失敗停止回数 | 3回 |
| 急変動停止条件 | 5分で±2%以上 |
| 再開方法 | 手動再開 |

## 9. 注文管理設計

### 9.1 注文ステータス

| ステータス | 内容 |
|---|---|
| CREATED | 注文作成前 |
| SENT | API送信済み |
| PARTIALLY_FILLED | 一部約定 |
| FILLED | 全約定 |
| CANCELED | キャンセル済み |
| FAILED | 注文失敗 |
| UNKNOWN | 状態不明 |

### 9.2 注文処理方針

#### 成行買い

```text
1. 注文前に残高確認
2. 最小注文金額確認
3. 成行買い注文送信
4. APIレスポンス保存
5. 約定確認
6. position作成
7. LINE通知
```

#### 指値売り

```text
1. 保有数量確認
2. 利確価格算出
3. 指値売り注文送信
4. APIレスポンス保存
5. 未約定の場合は監視
6. 約定時にpositionクローズ
```

#### 成行売り

```text
1. 損切り条件確認
2. 保有数量確認
3. 成行売り注文送信
4. 約定確認
5. positionクローズ
6. 損益計算
7. LINE通知
```

## 13. エラー処理設計

### 13.1 APIエラー

| エラー | 処理 |
|---|---|
| タイムアウト | リトライ |
| 認証失敗 | 即停止 |
| レート制限 | 一定時間待機 |
| 注文失敗 | DB記録、LINE通知 |
| 約定確認失敗 | 状態UNKNOWNとして再確認 |
| 残高不足 | 取引停止 |

### 13.2 DBエラー

| エラー | 処理 |
|---|---|
| 接続失敗 | システム停止 |
| INSERT失敗 | ログ出力、再試行 |
| 設定取得失敗 | 起動停止 |
| 注文記録失敗 | 注文実行前なら停止、実行後なら警告 |

重要：注文実行後にDB登録失敗した場合、実取引とDB状態がズレるため、必ずLINE通知する。

## 14. 手動操作設計

### 14.1 手動停止

DBの `bot_status.emergency_stop` を `1` にすることで、次回ループで自動売買を停止する。

```sql
UPDATE bot_status
SET emergency_stop = 1,
    reason = N'手動停止',
    updated_at = SYSDATETIME()
WHERE id = 1;
```

### 14.2 取引再開

```sql
UPDATE bot_status
SET emergency_stop = 0,
    is_trading_enabled = 1,
    reason = N'手動再開',
    updated_at = SYSDATETIME()
WHERE id = 1;
```

### 14.3 緊急全決済

保有ポジションがある場合、成行売りで全決済する。

```text
1. 保有ポジション確認
2. 保有数量確認
3. 成行売り注文
4. 約定確認
5. 損益計算
6. DB更新
7. LINE通知
8. 自動売買停止
```

## 20. 初期実装方針

初期実装では、以下を優先する。

```text
1. 確実に止まること
2. 注文履歴がDBに残ること
3. 売買理由がログに残ること
4. LINE通知で状況が分かること
5. 少額で本番検証できること
```

利益最大化のためのパラメータ自動調整、複数通貨対応、GUI、AI判断は後続フェーズとする。

## 21. 補足

自動売買システムでは、利益を出すロジックよりも先に、以下を確実に実装する必要がある。

```text
・誤発注しない
・暴走しない
・残高以上に注文しない
・異常時に止まる
・手動で止められる
・取引履歴が必ず残る
・実取引とDB状態のズレを検知できる
```

そのため、初期版では売買ロジックを単純にし、注文管理、リスク管理、ログ保存を優先する。
